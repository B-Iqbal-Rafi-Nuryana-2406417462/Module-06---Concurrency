# Commit 1 Reflection notes

Program diperbarui untuk dapat menerima dan menangani koneksi TCP dari browser dengan membaca data request HTTP yang masuk melalui method handle_connection. Proses pembacaan dilakukan secara efisien dengan membungkus TcpStream menggunakan BufReader, yang memungkinkan kita membaca data baris demi baris melalui method .lines(). Karena pembacaan jaringan dapat gagal dan mengembalikan tipe Result, .map(|result| result.unwrap()) digunakan untuk mengekstrak teks string secara langsung dari dalam iterator. Langkah yang sangat krusial di sini adalah penggunaan .take_while(|line| !line.is_empty()), yang memastikan program berhenti membaca stream ketika menemukan baris kosong. Hal ini sangat penting karena standar protokol HTTP menandai akhir dari sebuah header request dengan baris kosong tanpa instruksi ini, server akan terus menunggu masukan data dan menyebabkan proses di browser tertahan atau hang. Terakhir, .collect() digunakan untuk mengeksekusi iterator dan mengumpulkan semua baris tersebut ke dalam sebuah struktur data vektor Vec<_>. Melalui pendekatan ini, kita dapat mencetak dan mengamati format mentah dari request HTTP yang dikirimkan oleh browser, yang mencakup method seperti GET, path, versi HTTP, dan berbagai header terkait.

# Commit 2 Reflection notes

Fungsi handle_connection dikembangkan agar tidak hanya membaca request, tetapi juga memberikan balasan berupa halaman web ke browser. Modul fs ditambahkan untuk membaca seluruh isi berkas hello.html dan mengubahnya menjadi teks menggunakan fungsi read_to_string. Agar browser mengerti dan dapat merender halaman tersebut, kita harus membuat format response HTTP yang valid. Struktur response ini mencakup Status Line yaitu baris HTTP/1.1 200 OK yang menandakan bahwa koneksi dan request berhasil. Selanjutnya terdapat Content-Length yang memberi tahu browser ukuran pasti dari data yang akan diterima, sehingga browser tahu kapan proses pengunduhan data selesai. Sesuai protokol HTTP, kita juga menambahkan dua baris kosong sebagai pemisah antara bagian informasi header dengan isi konten HTML. Seluruh elemen di atas digabungkan menjadi satu kesatuan teks menggunakan makro format. Setelah format response siap, data tersebut diubah menjadi kumpulan byte dan dikirimkan kembali ke browser melalui aliran jaringan menggunakan perintah write_all.

# Commit 3 Reflection notes 

Server diperbarui agar dapat memvalidasi permintaan yang masuk dan memberikan respons secara selektif. Daripada memproses seluruh isi permintaan dari peramban, program kini dipersingkat untuk hanya membaca baris pertamanya saja guna mengenali rute yang dituju. Melalui percabangan logika bersyarat, kita memisahkan alur jenis respons. Jika pengguna meminta rute halaman utama yang valid, server akan memuat berkas halaman utama beserta baris status berhasil. Sebaliknya, jika pengguna mencoba mengakses rute lain yang tidak terdaftar, program akan beralih ke alur alternatif untuk mengirimkan halaman peringatan kesalahan beserta baris status tidak ditemukan.

Meskipun pemisahan alur respons tersebut berhasil berjalan dengan baik, implementasi awalnya menghasilkan penulisan langkah kerja yang berulang. Proses utama seperti pembacaan berkas fisik, kalkulasi panjang ukuran teks, dan pengiriman format balasan ke dalam aliran jaringan ditulis secara identik pada kedua alur logika tersebut. Untuk menyelesaikan masalah ini, kita melakukan proses penataan ulang struktur program untuk menyederhanakannya. Percabangan logika kini difokuskan untuk sekadar menentukan dua nilai yang pasti berbeda, yaitu teks baris status dan nama berkas sasaran. Setelah kedua sasaran tersebut ditetapkan, proses pembacaan teks dan pengiriman data hanya perlu dieksekusi satu kali di bagian akhir. Penataan ulang ini sangat dibutuhkan karena secara drastis mengurangi duplikasi instruksi, membuat program jauh lebih ringkas untuk dibaca, dan sangat mempermudah proses pemeliharaan di masa depan karena perubahan logika jaringan cukup dilakukan pada satu area saja.

# Commit 4 Reflection notes 

Simulasi dilakukan untuk melihat kelemahan utama dari arsitektur server dengan alur proses tunggal. Sebuah rute baru ditambahkan untuk sengaja menunda atau menghentikan sementara proses eksekusi program selama sepuluh detik. Ketika rute lambat ini diakses melalui satu jendela peramban dan rute utama diakses pada jendela peramban lainnya secara bersamaan, rute utama tidak dapat langsung dimuat. Peramban kedua harus menunggu hingga proses penundaan pada peramban pertama selesai sepenuhnya. Pengujian ini membuktikan secara langsung bahwa sistem alur tunggal sangat tidak efisien untuk melayani banyak pengguna secara serentak. Satu proses permintaan yang memakan waktu lama akan memblokir seluruh permintaan lain yang sedang mengantre di belakangnya, yang pada akhirnya akan menurunkan kinerja server secara drastis dan merusak pengalaman pengguna.

# Commit 5 Reflection notes

Server dikembangkan menjadi sistem yang mampu melayani banyak permintaan secara bersamaan dengan mengimplementasikan pola kumpulan pekerja atau thread pool. Pendekatan ini dipilih untuk mencegah pembuatan thread baru tanpa batas yang dapat menguras sumber daya server. Kita membatasi eksekusi hanya pada sejumlah thread pekerja yang sudah disiapkan sejak awal. Untuk mendistribusikan tugas penanganan koneksi kepada pekerja yang sedang menganggur, kita menggunakan saluran komunikasi antrean. Karena ujung penerima saluran ini harus dibagikan ke banyak thread pekerja secara bersamaan, kita melindunginya menggunakan kombinasi struktur memori Arc dan Mutex. Arc memungkinkan banyak thread memiliki referensi yang sama secara aman, sementara Mutex memastikan hanya ada satu pekerja yang bisa mengunci dan mengambil tugas dari antrean pada satu waktu tertentu. Terdapat sebuah detail sangat penting terkait siklus hidup penguncian Mutex ini, yaitu kita harus menggunakan perulangan tanpa henti dengan deklarasi variabel terpisah dibandingkan dengan konstruksi perulangan bersyarat. Penggunaan deklarasi variabel secara eksplisit memastikan sistem langsung melepaskan kunci antrean sesaat setelah tugas berhasil diambil, sehingga pekerja lain tidak terblokir dan tetap bisa mengambil antrean tugas berikutnya saat pekerja pertama sedang sibuk mengeksekusi tugasnya.

# Commit Bonus Reflection notes

Kita mengeksplorasi pembuatan fungsi alternatif bernama build untuk menggantikan fungsi bawaan new pada pembuatan kumpulan pekerja. Perbedaan utama dari kedua pendekatan ini terletak pada mekanisme penanganan kesalahan saat sistem diinisialisasi. Fungsi new pada implementasi sebelumnya dirancang untuk langsung memicu kepanikan dan menghentikan seluruh program secara paksa apabila menerima masukan jumlah pekerja sebesar nol. Pendekatan tersebut digunakan karena ketiadaan pekerja dianggap sebagai kesalahan fatal. Di sisi lain, fungsi build menggunakan pendekatan yang jauh lebih aman dengan mengembalikan sebuah tipe kembalian bernama Result. Jika masukan bernilai tidak valid, fungsi ini tidak akan merusak sistem secara tiba-tiba, melainkan mengembalikan sebuah objek peringatan kesalahan secara rapi. Hal ini memungkinkan bagian program yang memanggil fungsi tersebut untuk menangani kegagalan dengan lebih elegan, misalnya dengan menampilkan pesan kesalahan yang ramah pengguna atau beralih ke alur eksekusi cadangan. Perubahan kecil ini menunjukkan proses transisi yang penting menuju penulisan sistem peranti lunak yang jauh lebih tangguh untuk lingkungan produksi nyata.
